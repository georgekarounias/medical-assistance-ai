# Analyte extraction: the LLM maps, code copies, every value verified verbatim

Lab Reports come from many labs with different table layouts and bilingual (Greek/English) headers, so per-lab mapping templates would be a treadmill. Instead, an LLM looks at the extracted cell grid and answers only classification questions: which column is the analyte name / value / unit / reference range, and what is the canonical name for each row's analyte. Code then builds each Analyte Result by copying values from the cells the LLM pointed at, and rejects any row whose value, unit, or range does not appear verbatim in the source grid. The same safety architecture as ADR-0002: the LLM points, code copies — no number a doctor sees was ever generated by a model.

## Consequences

- Analyte rows are all-or-nothing per document ("tiered honesty"): Tier 1 (text chunks of the report) must succeed or the Ingestion is Failed; Tier 2 rows are stored only if every row verifies. If any row fails, zero rows are stored and the record carries `analytesExtracted: false` — trend queries never see a silently partial Panel, but the report remains searchable as text.
- Each Analyte Result keeps both `verbatimName` and `canonicalName`, plus provenance (page, table, row) into the stored PDF.
- Canonical naming starts as a curated mapping; formal coding (e.g., LOINC) is a later, additive step.
